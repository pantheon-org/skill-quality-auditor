# Plan: Remediate Audit Frontmatter & Validation Gaps

## Goal

Close the validation blind spot around `.context/audits/` files so that:

1. All audit `Analysis.md` and `Remediation.md` files carry **canonical, schema-valid** frontmatter (`title`, `type: audit`, `status: done`, `date`).
2. The pre-commit hooks validate audit files for frontmatter like any other `.context/` file.
3. The index regeneration script handles audit files **consistently** (no silent drops, no stale-variable false negatives, no miscategorisation into the `findings` group).
4. The `hk check` path detects stale indices involving audit file changes.

## Background

Audit files under `.context/audits/<skill>/<date>/` are generated by `reporter/analysis.go` and `reporter/remediation.go`. Frontmatter was added to both generators in commit `c1e70ba` ("fix: sync plan statuses…", PR #83). That commit also rewrote `regenerate-context-index.sh` (24 lines touched) and added the `.context/audits/**` exclusions to the two `hk.pkl` pre-commit hooks. Two pre-commit hooks (`context-frontmatter`, `context-index`) explicitly exclude `.context/audits/**`, creating a validation blind spot.

The canonical frontmatter the generators emit is:

- `Analysis.md`: `title: "Skill Audit — <skill>"`, `type: audit`
- `Remediation.md`: `title: "Remediation Plan — <skill>"`, `type: audit`

(`reporter/analysis.go:16` and `reporter/remediation.go:38`.) The `date` field is `r.Date` (the audit's run date), not the file's mtime.

### Critical review: the script bug is mischaracterised

The plan's original Step 4 / Open Q #3 characterised the `regenerate-context-index.sh` flaw as a "content-before-assign bug". **It is not.** The script does not crash with `UnboundLocalError` because sorted `.context/rglob("*.md")` yields `.context/analysis/*.md` files **before** any `.context/audits/*.md` file, so `content` is bound in those earlier iterations and survives into the audit iterations as a **stale value**. The skip condition on lines 46–47:

```python
if rel.startswith(".context/audits/") and not content.startswith("---\n"):
    continue  # skip legacy audits without frontmatter
```

checks the **previous** file's content, not the current file's. Verified empirically (the script runs to completion with `REAL_EXIT=1`, does not raise). The real consequences, observed by re-running the script:

| File | Behaviour | Why |
|------|-----------|-----|
| `adr-capture/2026-06-30/Analysis.md` | **Enters index with `type: finding`** (leaks into `findings` group) | prev file = `duplication-report-2026-04-29.md` (has `---`), skip is False; file has frontmatter but wrong `type` |
| `adr-capture/2026-06-30/Remediation.md` | Flagged as missing (correct, but for the wrong reason) | prev file = `adr-capture/Analysis.md` (has `---`) |
| `context-file/2026-06-30/Analysis.md` | **Silently dropped** | prev = `adr-capture/Remediation.md` (no `---`) |
| `context-file/2026-06-30/Remediation.md` | **Silently dropped** | prev = `context-file/Analysis.md` (no `---`) |
| `context-index/2026-06-30/Analysis.md` | **Silently dropped** | prev = `context-file/Remediation.md` (no `---`) |
| `context-index/2026-06-30/Remediation.md` | **Silently dropped** | prev = `context-index/Analysis.md` (no `---`) |
| `session-reflection/2026-07-03/Analysis.md` | **Silently dropped** ⚠️ despite having valid frontmatter | prev = `context-index/Remediation.md` (no `---`) |
| `session-reflection/2026-07-03/Remediation.md` | **Silently dropped** ⚠️ despite having valid frontmatter | prev = stale `adr-capture/Remediation.md` content (no `---`); Analysis was skipped so `content` was never updated to session-reflection/Analysis.md |

Two non-obvious, plan-relevant corollaries:

1. **Validly-frontmattered audit files are silently dropped today.** The `session-reflection` audits (generated by the current canonical generator) never reach the index. Removing the `hk.pkl` exclusions (Steps 2–3) without also removing the script's broken skip (Step 4) would leave them dropped. Steps 1, 2, 3, and 4 are **not independent** — they must land together.
2. **One audit file (`adr-capture/2026-06-30/Analysis.md`) has frontmatter but `type: finding`**, so it silently pollutes the `findings` group in `index.yaml`. Its title (`"Audit: adr-capture Analysis — 2026-06-30"`) also deviates from the canonical generator format. This file was hand-authored with a non-canonical pattern; the plan must fix it, not just the five missing-frontmatter files.

## Steps

### 0. Create a feature branch

Per the ways-of-working: branch off `main` with a `feat/` prefix before any edit.

```bash
git checkout -b feat/audit-frontmatter-validation main
```

### 1. Backfill canonical frontmatter on the 6 audit files

Use the **canonical generator format** (so a future `evaluate --store` regeneration produces byte-identical frontmatter) and preserve the historical `date: 2026-06-30` (backfilling frontmatter must not move the audit's run date).

`Analysis.md` pattern:

```yaml
---
title: "Skill Audit — <skill>"
type: audit
status: done
date: 2026-06-30
---
```

`Remediation.md` pattern:

```yaml
---
title: "Remediation Plan — <skill>"
type: audit
status: done
date: 2026-06-30
---
```

Apply to all six files listed above. For `adr-capture/2026-06-30/Analysis.md`, change `type: finding` → `type: audit` and the title to `"Skill Audit — adr-capture"` (keep `date: 2026-06-30`).

**Rejected alternative — re-run `evaluate --store`** (formerly Open Q #2): the generator stamps `date` with the *current* run date, so re-running would produce a new audit dated today and overwrite the historical 2026-06-30 snapshot, including any score deltas. Manual backfilling with canonical fields preserves history.

### 2. Remove `.context/audits/**` exclusion from `context-frontmatter` hook

In `hk.pkl`, delete `exclude = List(".context/audits/**")` from the `context-frontmatter` step (line 32). Audit files now carry frontmatter, so `validate-context-frontmatter.sh` should validate them against `context-frontmatter.schema.json` (whose `type` enum already includes `audit` — verified).

### 3. Remove `.context/audits/**` exclusion from `context-index` hook

In `hk.pkl`, delete `exclude = List(".context/audits/**")` from the `context-index` step (line 37). Changes to audit file frontmatter should trigger index regeneration.

### 4. Remove the broken audits-skip in `regenerate-context-index.sh`

Delete Python lines 46–47 (inside the heredoc, not shell lines) in `.agents/skills/context-index/scripts/regenerate-context-index.sh`:

```python
if rel.startswith(".context/audits/") and not content.startswith("---\n"):
    continue  # skip legacy audits without frontmatter
```

This **must** land in the same commit as Steps 1–3. With the stale-variable bug, valid audit files (e.g. `session-reflection`) are silently dropped and one audit (`adr-capture/Analysis.md`) is miscategorised as a `finding`. Removing the skip lets the existing `missing`/`parse_frontmatter` paths below it correctly flag frontmatter-less audit files rather than dropping them silently. After removal, there is no remaining use-before-assignment risk — `content = md.read_text()` (the next line) becomes the only definition site.

### 5. `check-context-frontmatter.sh`: no change required (verified)

The `check-context-frontmatter.sh` script (used by the `adr-frontmatter` hook) only tests that the first line is `---`. It has **no** audit-specific exclusion. Definitive: no edit needed. (The original plan hedged this as "if any".)

### 5b. `adr-undocumented` hook: no change required (verified)

The `check-undocumented-decisions.sh` script (used by the `adr-undocumented` hook) scans all `.context/**/*.md` files with **no** audit-specific exclusion. This is correct behaviour: audit files do not contain decision indicators (`## Decision`, `## Recommendation`, etc.), so they will not be flagged as undocumented decisions. The hook's `index.yaml` skip guard (line 62) also prevents false positives on files that reference the index. No edit needed.

### 6. Decide audit placement in the index

Removing the skip exposes audit files to the index. The script's `type_group_key` (line 77) has no `audit` entry:

```python
type_group_key = {"plan": "plans", "finding": "findings", "analysis": "analysis", "instruction": "instructions"}
```

Files with `type: audit` fall through to the `other` bucket (lines 83–93). That is structurally valid but produces an `other:` section whenever audits exist — likely surprising for readers expecting `audits:`.

**Decision (resolving former Open Q #1):** add an explicit `audits` section. Edit `regenerate-context-index.sh`:

- Add `"audit": "audits"` to `type_group_key`.
- Add `"audits"` to `type_order` (after `"analysis"`, before `"instructions"`, or at the end depending on desired reading order — recommend after `analysis`).
- Add `"audits": "Audits"` to `type_label`.

Rationale: audit files are first-class generated artifacts and are now validated like all other `.context/` files; surfacing them under a named `audits:` section is consistent with the schema's `audit` enum value and avoids the catch-all "other" bucket. The alternative (accept `other`) is rejected because "other" is a fallback for unknown types, not a deliberate category.

If, on review, audits are deemed noise in the index, the alternative is to **skip them in the index deliberately** with an explicit, correctly-scoped guard:

```python
if rel.startswith(".context/audits/"):
    continue  # generated artifacts — not indexed
```

placed **after** `content = md.read_text()` (so it never reads stale state) and accompanied by a comment. Pick one path before implementing; this plan adopts the "add `audits` section" path.

### 7. Regenerate `.context/index.yaml`

```bash
.agents/skills/context-index/scripts/regenerate-context-index.sh
```

Inspect the diff: expect a new `audits:` section with 8 entries, the `adr-capture/Analysis.md` entry **moved out of `findings:`** (it previously leaked there as `type: finding`), and no `other:` bucket.

### 8. Verify

```bash
hk check          # pre-commit hooks — incl. frontmatter validation on audit files
go test ./...     # reporter + scorer tests; regen-script changes are shell, not Go, but run anyway per AGENTS.md
hk run pre-push   # full pre-push gate incl. binary build, duplication, batch, structural eval
```

Risk note: `hk check` now exercises `validate-context-frontmatter.sh` against audit files. The schema enum includes `audit`, so `type: audit` is accepted; `status: done` is in the enum; `date` matches `^\d{4}-\d{2}-\d{2}$`. This is the **first time audit files are schema-validated** — if any file has trailing whitespace in a YAML value or similar subtle issue, `hk check` will now flag it.

Coverage gap — `reporter/*_test.go` does not assert on specific frontmatter fields. `remediation_test.go` checks `strings.HasPrefix(plan, "---\n")` only; `analysis_test.go` has zero frontmatter coverage. A future regression that drops the canonical `type: audit` or `status: done` line would not be caught. Adding frontmatter assertions to both test files is recommended for durability but is out of scope for this plan's immediate remediation.

## Open Questions

_None remaining._ The three original open questions are resolved below for the record:

- **Q2 — Re-run `evaluate --store` instead of hand-editing?** → **Resolved (Step 1):** no, manual backfill with canonical fields, to preserve the historical 2026-06-30 audit date and score snapshot. Re-running would restamp `date` to today and overwrite the original result.

## Sequencing & risk summary

| Risk | Mitigation |
|------|------------|
| Removing `hk.pkl` exclusions before backfilling frontmatter breaks the next commit on any audit file | Steps 1–4, 6–7 land atomically in one commit |
| Removing the script skip alone (without Step 6) pollutes the index with an `other:` bucket | Steps 6–7 land in the same commit |
| `hk check` (`context-index` hook via `--check`) detects stale index and would fail mid-commit if Step 7 is left out | Commit includes Step 7 (regen index), so the index is fresh after the commit |
| `validate-context-frontmatter.sh` rejects audit files over a non-`audit` `type` | Step 1 sets `type: audit` on all six files; schema enum confirmed to include `audit` |
| Score drift if anyone re-runs `evaluate --store` after the fix | Step 1's frontmatter is canonical; re-running produces byte-identical frontmatter except `date`, so it is still discouraged for historical audits |
